FoundrAI — System Design

Autonomous AI Advisory Platform for Startup Founders

Backend: FastAPI + Python 3.12 AI: Groq + Llama 3.1 / 3.3 RAG: pgvector + sentence-transformers DB: Supabase PostgreSQL 15 Frontend: React 18 + Vite + TailwindCSS Deploy: Render + Vercel

1 What FoundrAI Does

FoundrAI is a private AI analyst for startup founders. A founder uploads their financial data (CSV, Excel, PDF) and asks natural-language questions. Four specialist AI agents answer in sequence, each reading the previous agent's output — producing structured analysis grounded in the founder's own data plus live market news.

User Journey

  1. Register / Login

    Supabase Auth creates a JWT-secured session. Email + password with optional TOTP MFA (6-digit authenticator app).

  2. Upload Financial Document

    Drag and drop a CSV, Excel, PDF, DOCX, or image. The backend extracts text, detects financial columns (revenue, burn, CAC, LTV), chunks the document, embeds it into pgvector, and saves structured metrics as a JSON snapshot.

  3. Ask a Question

    e.g. "What is my runway if CAC increases 20% next quarter?" The question goes to the 4-agent pipeline. Agents read the founder's document chunks (RAG retrieval) and answer in sequence, streaming each result back via SSE.

  4. Run a Monte Carlo Simulation

    Choose bear / base / bull scenario, set CAC and burn change sliders, pick 1–24 month horizon. 10,000 paths run in ~100ms. Returns P10/P50/P90 revenue bands and runway estimates.

  5. View Charts

    Recharts AreaChart renders the three confidence bands with gradient fills. Historical simulations stored in Supabase for comparison.

5 Authentication & Security

How JWT Auth Works

Supabase issues HS256 JWTs signed with SUPABASE_JWT_SECRET. The backend validates every token using PyJWT in the verify_jwt FastAPI dependency — injected via Depends(verify_jwt) on every protected route. Token claims include sub (the founder UUID) which scopes all DB queries.

EndpointMethodAuth RequiredDescription
/auth/registerPOSTNoCreate Supabase user + founders row (service role)
/auth/loginPOSTNosign_in_with_password → return JWT tokens
/auth/refreshPOSTNoExchange refresh token for new access token
/auth/logoutPOSTNoInvalidate session (best-effort)
/auth/mfa/enrollPOSTYesBegin TOTP enrollment — returns QR code URI
/auth/mfa/verifyPOSTNoVerify 6-digit TOTP, upgrade session to AAL2
/auth/meGETYesReturn current founder profile from JWT claims

Key Security Decisions

  • Two Supabase clients: anon key for auth operations (sign_up/sign_in), service role key for backend DB writes that bypass RLS. Service role key never sent to browser.
  • Row Level Security (RLS) on every table: auth.uid() = founder_id. Even if someone got the anon key, they can only read their own rows.
  • Parameterised queries only — Supabase Python client uses prepared statements internally. No string concatenation in SQL.
  • No secrets in logs — error messages never include tokens, keys, or stack traces. Global exception handler returns a generic message.
  • CORS — restricted to origins in CORS_ORIGINS env var. Never * in production.
  • Input validation — all request bodies are Pydantic models. Fields have types, ranges, and allowed values enforced before any business logic runs.

6 4-Agent AI Pipeline

Agents run sequentially, not in parallel. Each agent reads the previous agent's structured JSON output plus the RAG-retrieved context. This chaining lets each agent reason at a higher level than if it ran independently.

The orchestrator runs Groq calls in a ThreadPoolExecutor (Groq's Python client is synchronous/blocking) and wraps each call with asyncio.get_event_loop().run_in_executor() to keep FastAPI's event loop free. Between each agent, an SSE event is yielded so the frontend updates in real time — users see answers appearing agent by agent.

SSE Event Stream Format

rag_context
Fired first — contains the retrieved document chunks that agents will read
agent_update
Fired after each agent completes — contains agent_name, JSON content, timestamp
final
Pipeline complete — contains list of agents_ran and rag_chunks_used count
error
Unrecoverable stream error — agent failures are non-fatal and do not emit this

Failure Resilience

Each agent call is wrapped in try/except. If one agent fails (e.g. Groq timeout), the orchestrator logs the error, emits an agent_update with an error payload, and continues to the next agent. Partial results are better than no results. The final event is always emitted.

7 RAG Pipeline

Why RAG?

Founder financial documents can be hundreds of rows of CSV or dozens of pages of PDF. Sending the entire document to the LLM every query is expensive and slow. RAG retrieves only the 5 most relevant passages for each question, keeping prompts short and responses grounded in actual data.

Indexing (called after every upload)

  1. Text extraction

    extractors.py converts the raw file bytes to plain text. CSV/Excel → pandas, PDF → pdfminer, DOCX → python-docx, images → pytesseract OCR, plain text → direct read.

  2. Chunking

    indexer.py splits text into overlapping 512-token chunks (with stride so context isn't lost at chunk boundaries).

  3. Embedding

    encoder.py passes each chunk through sentence-transformers (all-MiniLM-L6-v2). Output: 384-dimensional float vector per chunk.

  4. Storage

    Each chunk + vector stored in document_embeddings table with founder_id, source filename, doc_type, and chunk_index.

Retrieval (called before each query)

  1. Embed the question

    Same encoder as indexing time — question gets a 384-dim vector.

  2. pgvector cosine search

    retriever.py runs: SELECT ... ORDER BY embedding <=> query_vector LIMIT 5 — restricted to rows where founder_id matches the authenticated user.

  3. Format context

    pipeline.chunks_to_context() formats chunks as a numbered list: [1] (source, score=0.87): text... This is injected into every agent's system prompt.

8 Monte Carlo Simulation Engine

What It Models

Given a starting MRR, burn rate, and cash balance, the engine simulates 10,000 possible futures. Each path draws a random monthly growth rate and a noisy burn rate, compounds them forward month-by-month, and tracks when (if ever) cumulative burn exceeds available cash.

Implementation — Vectorised NumPy

All 10,000 paths run simultaneously using NumPy matrix operations — no Python loop per simulation. Shape: (10000, months_ahead) matrices for growth rates and burn noise. np.cumprod compounds revenue. np.cumsum accumulates burn. np.argmax finds the runway month. Total runtime: ~100–200ms.

Scenario Parameters

ScenarioMonthly growth mean (μ)Standard deviation (σ)Interpretation
Bear-1% per month±5%Slow decline, high volatility
Base+5% per month±4%Moderate growth
Bull+12% per month±6%Strong growth, high variance

CAC & Burn Adjustments

  • CAC change: drags growth mean by cac_change × 0.3. A 20% CAC increase reduces expected monthly growth by 6%.
  • Burn change: scales the base burn rate linearly before adding Gaussian noise.
  • Burn volatility: ±8% of monthly burn (Gaussian noise), clipped to 50%–200% of base to prevent unrealistic extremes.

Data Seeding from Uploads

The /simulate router reads initial_metrics from the selected upload row. This JSON snapshot contains revenue (MRR), burn_rate, cac, and ltv extracted by extractors.py from the founder's CSV at upload time. If no upload is selected, the engine falls back to hardcoded conservative defaults.

9 News Ingestion Pipeline

Components

NewsCatcher API
Article Discovery

Searches 6 startup-relevant topics: "startup funding", "SaaS growth", "venture capital", "startup failure", "product market fit", "CAC LTV SaaS". Returns up to 50 article URLs per run.

News-Please
Full-Text Extraction

Scrapes the full article body from each URL (10s timeout). Returns title, text, author, publish date. Articles with no maintext are skipped.

APScheduler
Background Scheduler

Runs ingest_news_batch every 6 hours in a background thread. Started in the FastAPI lifespan on app startup. Disabled if NEWSCATCHER_API_KEY is not set.

Deduplication
URL-Based Dedup

Before scraping, checks if the article URL already exists in news_articles. Skips duplicates — prevents re-indexing the same article across runs.

Ingestion Log

Every run writes to ingestion_logs: ingested count, skipped duplicates, error count, duration in seconds. Useful for monitoring the health of the news pipeline.

10 Storage Layer

supabase_client.py — Singleton Pattern

get_supabase_client() returns a cached Supabase client using the service role key. This client bypasses RLS and is only used server-side. Called via Depends(get_supabase_client) in routers or directly in background jobs. The connection is warmed at startup in the lifespan context manager.

gcs_client.py — GCP Cloud Storage

Raw uploaded files are stored to gs://{GCS_BUCKET_NAME}/uploads/{founder_id}/{filename}. The GCS path is stored in the uploads table. In local dev without GCP credentials, the upload still processes — GCS storage is skipped with a warning log.

extractors.py — Multi-Format Text Extraction

File TypeLibraryFinancial Column Detection
CSV / ExcelpandasYes — detects revenue, burn_rate, cac, ltv, mrr, arr, growth_rate by header name matching
PDFpdfminer.sixNo — full text extraction only
DOCXpython-docxNo — paragraph text extraction
Images (JPG, PNG, WEBP)pytesseract (Tesseract OCR)No — OCR text only
Plain textdirect read (UTF-8)No

For CSV/Excel with financial columns, extractors.py produces an initial_metrics JSON object — a summary of the first detected values for each metric. This seeds the Monte Carlo simulation.

QueryPage — SSE Streaming

The QueryPage uses streamQuery() which calls fetch() with { method: "POST", body: JSON, headers: { Authorization: Bearer ... } } and reads the response body as a ReadableStream. Each event: / data: pair is parsed and the callback is invoked. This is SSE without EventSource (which doesn't support POST requests). An AbortController is used to cancel the stream if the user navigates away.

SimulatePage — Recharts AreaChart

Three <Area> series (P10, P50, P90) with linearGradient fills defined in SVG <defs>. P10 gets dark terracotta fill, P50 gets main terracotta fill, P90 gets green fill. Custom TooltipContent component formats values as $K or $M.
